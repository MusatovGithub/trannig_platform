# AGENTS.md

Этот файл задает общие инструкции для AI-агентов в репозитории `trannig_platform`.
Правила применяются ко всем задачам, если пользователь явно не указал иное.

## Контекст проекта

- Проект работает в продакшене: приоритет - стабильность и обратная совместимость.
- Стек: Python 3.12, Django 5.1, Celery, Redis, PostgreSQL.
- Базовые настройки Django: `config.settings.local` (для локальной разработки).
- Основные доменные приложения: `achievements`, `authen`, `competitions`, `custumer`, `employe`, `groups_custumer`, `group_classe`, `market`, `news`, `supervisor`.

## Обязательные принципы

1. Ничего не ломать: минимальные, точечные изменения без лишнего рефакторинга.
2. Не менять публичное поведение без явного запроса пользователя.
3. Сохранять текущие контракты моделей, URL и шаблонов.
4. Не удалять и не переименовывать файлы/символы без необходимости.
5. Не трогать секреты и чувствительные данные (`.env`, ключи, токены, пароли).

## Правила работы с кодом

- Перед изменениями изучать связанный код (документацию(docs/), модели, вью, urls, шаблоны, формы, миграции, тесты).
- Для багфиксов предпочитать самый узкий фикс в месте проблемы.
- Для новой функциональности опираться на существующие паттерны проекта.
- Соблюдать стиль проекта (PEP8, текущие соглашения по именам и структуре).
- Не смешивать в одном изменении функциональные правки и большой форматинг.
- В локальной разработке источник стилей и фронтенд-ассетов: `static/` (а не `static_cdn/`).

## Работа с моделями и БД

- Любые изменения моделей делать максимально осторожно.
- Если меняется схема БД, явно описывать риск и влияние на существующие данные.
- Не выполнять опасные миграционные действия без подтверждения пользователя.
- Не удалять поля/таблицы, используемые продакшен-логикой, без согласования.

## Проверки перед завершением задачи

Минимальный чеклист:

1. Локально проверить измененные участки кода.
2. Убедиться, что импорты корректны и нет синтаксических ошибок.
3. Проверить линтер только для затронутых файлов (если применимо).
4. Для изменений бизнес-логики добавить или обновить тесты (если в проекте есть покрытие для этой зоны).
5. В ответе кратко указать риски и что именно проверено.
6. Создать документацию с описание функционала, для сохранения контекста и понимания работы фитчи, улучшения или не повторения бага в будущем.

## Ограничения на рискованные действия

- Не выполнять разрушительные git-команды (`reset --hard`, `checkout --`, массовые удаления) без явного разрешения.
- Не менять инфраструктурные и деплой-настройки без отдельного запроса.
- Не обновлять зависимости "заодно", если задача не про зависимости.
- Не выполнять массовые автоматические правки по всему проекту без необходимости.

## Формат ответа пользователю

- Писать кратко и по делу.
- Сначала результат, затем список измененных файлов.
- Отдельно отмечать потенциальные риски и шаги проверки.
- Если не хватает контекста, задавать уточняющий вопрос до внесения изменений.

## Приоритеты принятия решений

Если есть конфликт между "быстрее" и "безопаснее", выбирать "безопаснее".
